# Расширение: несколько точек выдачи

## Что уже сделано

- **Сущность `Point`** (таблица `points`): название, адрес/комментарий, мягкое удаление.
- **Меню «Точки выдачи»** — раздел рядом с журналом прокатов: список точек, добавление, редактирование, удаление.
- **Связи:**
  - **User.point** (nullable) — к какой точке привязан сотрудник. `null` = без привязки (админ может видеть все).
  - **Rental.point** (nullable) — в какой точке оформили прокат. `null` = старая запись до введения точек.
- При первом запуске создаётся точка по умолчанию **«Основная»**.

Текущее поведение **не ломается**: старые прокаты и сотрудники без точки работают как раньше (поле nullable).

---

## Что доделать для «у каждой точки свой журнал»

### 1. Журнал прокатов по точкам

- **Вариант A (простой):** на странице `/rentals` добавить фильтр «Точка» (выпадающий список). Админ/суперадмин выбирает точку и видит только прокаты этой точки. По умолчанию — «все» или «текущая точка пользователя».
- **Вариант B (отдельные страницы):** в меню «Точки выдачи» у каждой точки — кнопка «Журнал» → переход на `/rentals?pointId=1`. На `/rentals` читать `pointId` из запроса и отдавать в сервис: `rentalService.findActive(pointId)` и т.д.

Реализация:

- В **RentalRepository** добавить методы с фильтром по точке, например:
  - `findByPointIdAndStatusOrderByDateFromDesc(Long pointId, RentalStatus status)`
  - и по списку статусов: `findByPointIdAndStatusInOrderByDateFromDesc(Long pointId, Collection<RentalStatus> statuses)`.
- В **RentalService** — перегрузки `findActive(pointId)`, `findCompleted(pointId)` и т.д. (если `pointId == null` — без фильтра, как сейчас).
- В **RentalController** на `GET /rentals` брать `@RequestParam(required = false) Long pointId` и передавать в сервис. Для WORKER с `user.getPoint() != null` подставлять их точку (или показывать только её журнал).

### 2. Привязка сотрудников к точке

- На странице **«Штат сотрудников»** в форме добавления/редактирования пользователя добавить поле **«Точка»** (выпадающий список из `pointService.findAllActive()`).
- При сохранении выставлять `user.setPoint(point)` или `null` (без привязки).
- Логика доступа:
  - **WORKER**: если `user.getPoint() != null`, в журнале прокатов показывать только прокаты этой точки (и при создании проката подставлять `rental.setPoint(user.getPoint())`).
  - **ADMIN / SUPER_ADMIN**: точка в профиле опциональна; в журнале — фильтр по точке или «все».

### 3. При создании проката подставлять точку

- В форме оформления проката (или на бэкенде) задавать **точку**:
  - для WORKER с привязкой: автоматически `rental.setPoint(currentUser.getPoint())`;
  - для ADMIN/SUPER_ADMIN: либо выбор точки в форме, либо по умолчанию одна точка (например, первая из списка).
- В **RentalService.createRental** добавить параметр `Long pointId` (или брать из текущего пользователя в контроллере и передавать), загружать `Point` и вызывать `rental.setPoint(point)`.

### 4. Оборудование и точки (опционально)

- Сейчас техника общая (одна база). Если нужно, чтобы в каждой точке был свой «склад», можно добавить **Equipment.point** (nullable): `null` = общий склад, иначе техника привязана к точке. Тогда при выдаче в точке показывать только технику этой точки или общую. Это уже следующий шаг после того, как журнал и сотрудники по точкам заработают.

## Итог

Расширение **потянем без поломок**: точки добавлены, связи в БД есть, меню и CRUD точек готовы. Дальше — фильтрация журнала по точке, привязка сотрудников к точке в форме штата и подстановка точки при создании проката. Всё это делается небольшими шагами поверх текущего кода.
